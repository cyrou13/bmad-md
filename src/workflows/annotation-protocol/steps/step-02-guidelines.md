# Step 2: Design Annotation Guidelines

## Agent

**Daria** (Data Governance Officer)

## Objectives

- Create comprehensive annotation instructions
- Provide visual examples for each category
- Document handling of edge cases

## Instructions

### 2.1 Structure Annotation Guidelines

```markdown
# Annotation Guidelines v[X.X]

## 1. Overview
- Purpose of annotations
- How annotations will be used
- Importance of consistency

## 2. Getting Started
- Tool overview
- Workflow steps
- Saving and submission

## 3. Annotation Instructions

### 3.1 [Task Type] Instructions
- Step-by-step process
- Technical requirements (e.g., bounding box margins)
- Quality expectations

### 3.2 Label Definitions
[For each label:]
- Definition
- Inclusion criteria
- Exclusion criteria
- Visual examples (positive, negative, borderline)

## 4. Special Cases
- Ambiguous findings
- Multiple findings
- Poor quality data
- Artifacts

## 5. Common Errors
- [Error 1]: Description and how to avoid
- [Error 2]: Description and how to avoid

## 6. FAQ
- Frequently asked questions and answers
```

### 2.2 Visual Examples

For each label category, provide:

**Positive Examples (3-5):**
- Clear, unambiguous cases
- Varying presentations
- Different severities

**Negative Examples (3-5):**
- Similar-looking but different conditions
- Normal variants that could be confused
- Artifacts mimicking findings

**Borderline Examples (2-3):**
- Edge cases with correct classification
- Explanation of decision rationale

### 2.3 Annotation Tool Instructions

Document tool-specific guidance:
- Screenshot-based tutorials
- Keyboard shortcuts
- Quality checks before submission
- Error correction procedures

### 2.4 Version Control

Establish versioning:
- Guidelines version numbering
- Change log
- Annotator notification process for updates

- [ ] Annotation Guidelines document created
- [ ] Visual examples curated
- [ ] Tool instructions documented
- [ ] Version control established
